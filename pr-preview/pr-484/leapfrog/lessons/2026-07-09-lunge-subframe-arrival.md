# 2026-07-09 — Lunge fix: "runs before the arrival check" beats "corrects next frame"; test the sub-frame cell

**Did / learned.** Codex caught a real corner in the merged lunge controller (#313): it set `vrel` for the NEXT integration step, so its correction always landed one frame late — invisible on long flights, but a flight SHORTER than one frame (nearest-organ clamp `targetRel≈4` → 2.5m at 20fps) arrived on frame 1 under constant speed and frame 2 under the profile: an arrival-frame LAW break in exactly the low-FPS/near-organ case. Fix: make position tracking LITERAL — the lance branch runs BEFORE the arrival check in the same loop iteration, so overwrite `s.rel` with the profile's analytic position (antiderivative + constant continuation) every frame; `vrel` now carries only the instantaneous profile speed for the homing read. Arrival is then identical to constant speed for EVERY dt by construction, including sub-frame flights. T-W8 gained the honest cell: `rel 4 @ 1/20`, asserting both profile==constant AND that the cell really is a frame-1 arrival.

**→ Systematize.** (1) A "corrects on the next step" controller has a one-step blind spot at every boundary — if a same-iteration hook exists BEFORE the check you must satisfy, write the state directly there instead of steering the next step. (2) An invariance sweep needs the DEGENERATE cell (flight ≤ one frame, the config clamp minimum × the worst dt) — long-flight cells can all pass while the boundary breaks. (3) When asserting a boundary cell, also assert the cell is honest (`ref === 1`) so a future tuning change can't silently turn it into a long-flight cell.

**→ Leapfrog.** The lunge is now provably arrival-exact everywhere; the same literal-position-tracking shape is the template for any future "follow a curve, land on the same frame" mechanic (boss lunges, reflected-amber returns).
