# Drowned Forum — killing the copy-paste read: the z-axis size-octave + the mirrored-build variant (Fable 4.2 PASS)

**What we did.** Owner, on the live preview after the two-shelf walls shipped: "I still see a lot of those TREE props and the STEP props repeating next to each other" — and, separately, a SCALE hunch: "aren't the rail columns usually really tall? if the walls are so big aren't the trees meant to be bigger?" Fable pre-assessed; we built its exact plan (a shared size-octave + a mirrored rail variant) and passed the gate 4.2 (trees 4.4, rail 4.1).

**THE OWNER'S SCALE HUNCH WAS HALF RIGHT — and the split is the lesson.** A three-register DEPTH LADDER (street rail h15-20 → trees → monument walls h28-53) is what makes the monuments read huge; the GAP between registers is the cue. So: the rail's small height is CORRECT (a real street portico was ~5-9m against 30m+ basilica walls — raising it flattens the hierarchy and the walls stop reading monumental). But the TREES were genuinely under-scale (uniformly h12-16, below the rail) — promoting ~25% to mature-pine h15-20 was the real fix. **When an owner says "X looks too small," don't reflexively scale X — check which REGISTER is misplaced. Often the fix is a sibling, and scaling the flagged prop would collapse a gap that's doing load-bearing work.**

**REPETITION IS A VARIANCE PROBLEM, NOT A COUNT PROBLEM — and thinning doesn't fix it.** We'd already thinned pinisle/drumfall (Fable passed it "singular"), yet the owner still read clones, because the surviving instances share a shape, a size, and a facing. Density controls how MANY; variance controls whether the ones you see read as clones. They're orthogonal knobs.

**THE SIZE-OCTAVE, PORTED + PER-AXIS.** The Mire already had `sizeOctave` (Fable-88 lever-6: a pure per-instance hash, `side` XOR'd in so the two flanks decorrelate, render-only → determinism-safe) but only in the `bi===4` branch. Ported it to the forum's `bi===0` branch as a shared mechanism + a new `octaveAxis:'z'` flag:
- **Uniform** (pinisle): `k *= bucket` scales the whole tree → a parent/child grove (mother/mid/child) while the 1.25r ASPECT COUPLING stays locked (the aspect IS the tree's name-test; raise the base coupling and you get a taller CLONE, not a grove — the octave gives spread WITHOUT touching the ratio).
- **Z-only** (viamarina): `octZ = bucket`, applied as `sclV.z = d.r*k*octZ` in the compose — scales only the segment LENGTH (intercolumniation ±20%), leaving height + footprint-x + lane clearance untouched. "One city, one column order; segments differ by how much street SURVIVED, not by column size." The compose already had per-axis multipliers (`hK`, `giantH`) so this was a 1-line thread, not a new system.

**A LIT-MATERIAL MIRROR MUST BE GEOMETRY-LEVEL, NOT A SCALE SIGN.** The rail's other clone tell was the collapse gradient always falling the same way. No rigid transform fixes it: a negative-z instance scale (`sclV.z = -…`) flips triangle winding + normals → backfaces cull out and lighting inverts on the lit `MeshStandardMaterial`. The fix is a parameterized `buildViamarina(zSign)` that NEGATES z-POSITIONS (and the drum rolls) — each part is merely RELOCATED, no negative scale, so winding/normals stay valid — appended as a second archetype `viamarinaM` at the END of ARCHETYPES (determinism law: never reorder slots). Both A+B at step 46 (combined = the old single-stream 23) so the two salted park-hashes interleave and the collapse direction alternates down the run.

**RANDOMIZE THE RUIN, NEVER THE ARCHITECTURE — the anti-scatter guardrail.** The forum's identity is ORDER; the failure direction is the "weird stone things" scatter we killed once already. So HOLD constant: column girth, the h15-20 datum (level architrave fragments = the "this was BUILT" cue), floor 0.30 (causeway continuity), yaw ≤ ~0.18 on architecture, every baked waterline level. VARY only: surviving segment length, collapse direction, per-instance survival, tree size + yaw. Past yaw ~15° or unlocking height is the tipping point back into scatter.
